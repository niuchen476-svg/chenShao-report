BACKGROUND · 背景与痛点
为什么需要
三个智能体协同？
01
信息分散，人工整合低效
赛事数据、舆情热点、AI预测来自不同数据源，人工每日汇总耗时2小时以上，且容易遗漏关键信息。
02
单一Agent能力有限
一个Agent同时处理API调用、舆情分析、内容生成，任务耦合导致失败率高，难以定位问题根源。
03
无法7×24小时自动运行
依赖人工触发的信息推送，无法保证每日定时交付，遇到网络异常或人员变动时系统立即失效。
解决方案：借鉴企业微服务架构思想，将复杂任务拆解为三个职责单一、独立运行的ArkClaw实例，通过TOS对象存储实现异步解耦协作，构建7×24小时自动化生产流水线。

WORKFLOW · 执行流程
每日自动化生产流水线
01
数据采集触发 Claw-A Claw-B
每天07:50，Claw-A自动调用football-data.org API拉取最新赛程比分，同时抓取B站世界杯热点评论；Claw-B并行执行AI胜率预测分析。
ArkClaw定时任务 → Skill执行 → football-data.org + B站评论 → 结构化JSON
02
TOS握手信号 Claw-A TOS存储桶
A/B将处理好的数据写入TOS存储桶后，额外写入ready.json作为握手信号，同时向飞书群发送「✅ 数据已就绪」消息。
TOS.put("matches.json") → TOS.put("ready.json", {status:"ready"}) → 飞书通知
03
智能体握手响应 Claw-C
Claw-C每5分钟轮询TOS，检测到ready.json后立即响应，向飞书群回复「🤝 收到，晨哨启动中...」，模拟企业协作中的任务接收确认机制。
TOS.get("ready.json") → status==="ready" → 飞书回复握手 → 触发生成任务
04
数据整合与分发 Claw-C
Claw-C读取A/B写入的所有数据，调用大模型进行AI分析生成预测文字，整合为标准data.json，同步推送到GitHub Pages更新网页，并向飞书群推送完整晨报。
TOS.read() → AI分析 → data.json → GitHub API push → 飞书晨报推送 → TOS.delete("ready.json")

WHY TOS NOT 飞书?
个人版飞书机器人只能发消息，无法订阅群消息事件。TOS作为共享存储介质，实现Agent间的异步解耦通信，这正是企业级微服务架构的标准模式。

TECH STACK · 技术栈与企业对标
对标企业级多Agent架构
ENTERPRISE COMPARISON
  • 任务分解：将复杂业务拆解为数据采集、分析预测、整合分发三个独立职责，对应企业中数据工程师、分析师、运营的协作分工
  • 异步解耦：Agent间不直接通信，通过TOS共享存储传递数据，降低耦合度，任一Claw故障不影响其他模块
  • 信号驱动：ready.json握手机制模拟企业消息队列（如Kafka/RabbitMQ）的事件驱动模式，实现生产者-消费者解耦
  • 可观测性：飞书群全程记录三Claw的握手、启动、完成消息，实现生产流程的可视化监控
  • 弹性扩展：新增数据源只需接入新Claw，写入TOS即可，无需修改现有Claw逻辑
